📘 Component Specification (V2)
Component Name: Student Program Ledger (SPL)
________________


🎯 1. Purpose
The Student Program Ledger (SPL) manages and displays the education program value, Monthly Education Credits (MEC), tuition usage, and refund lifecycle for each enrolled student under the Education Easy-Pay Investment Plan (EEPIP).
This component ensures:
* Transparent tracking of program value
* Controlled monthly value usage (MEC)
* Proper refund eligibility enforcement
* Full auditability of student financial interactions
________________


🧩 2. Core Concept
* The system maintains a fixed program value (Rs. 1,800,000)
* Each month:
   * MEC is credited
   * Tuition is debited
* Net balance remains constant
Additionally:
* A time-bound cooling-off window allows controlled refund handling
* All actions are consent-driven and auditable
________________


📊 3. Business Rules
________________


🔹 Rule 1: Initial Program Value
On enrollment:
program_value = 1,800,000 LKR
________________


🔹 Rule 2: Monthly Education Credits (MEC)
MEC = program_value × 16% ÷ 12 = 24,000 LKR
Each month:
* Credit: MEC (24,000)
* Debit: Tuition (24,000)
________________


🔹 Rule 3: Balance Integrity
closing_balance = opening_balance
________________


🔹 Rule 4: Duration
* 36 months (BSc phase)
________________


🔹 Rule 5: Non-Cash Nature
* Program value and MEC:
   * ❌ Not withdrawable
   * ❌ Not transferable
   * ✔ Represent education entitlement only
________________


🔹 Rule 6: Cooling-Off Policy
* Duration: 14 days from payment_date
* Refund allowed ONLY IF:
today <= cooling_off_end_date AND today < program_start_date
________________


🔹 Rule 7: Program Commencement
Defined as:
* First class OR
* LMS access OR
* Orientation session
________________


🔹 Rule 8: Refund Restrictions
* After:
   * Cooling-off expiry OR
   * Program commencement
👉 Refund = NOT allowed
________________

⚙️ 5. System Behavior
________________


🔹 On Enrollment
* Create StudentProgramAccount
* Set:
   * program_value = 1,800,000
   * payment_date = today
   * cooling_off_end_date = +14 days
   * status = ACTIVE
________________


🔹 Monthly Scheduler
For each ACTIVE account:
1. Check:
   * sequence_no < 36
   * academic_status = ACTIVE
2. Generate:
   * MEC entry (+24,000)
   * Tuition entry (–24,000)
________________


🔹 Refund Request Flow
________________


Step 1: Request
* Student submits request
* Set:
   * refund_status = REQUESTED
________________


Step 2: Eligibility Check
IF today <= cooling_off_end_date AND today < program_start_date
   THEN eligible = TRUE
ELSE
   eligible = FALSE
________________


Step 3: Calculation
admin_fee = program_value × admin_fee_rate
refundable_amount = program_value - admin_fee - incurred_costs
________________


Step 4: Decision
* If eligible → APPROVED
* Else → REJECTED
________________


Step 5: Processing
* Refund issued
* refund_status = PROCESSED
* account.status = DROPPED
________________


🔹 On Dropout
* status → DROPPED
* Stop MEC generation
________________


🔹 On Completion
* status → COMPLETED
________________

⚙️ System Behavior Updates
________________


🔹 Payment State Machine
INITIATED → CONFIRMED → LOCKED
Rules:
* INITIATED
   * Payment started but not verified
   * ❌ No seat allocation
   * ❌ No MLM activation
* CONFIRMED
   * Payment verified by bank
   * ✔ Seat can be reserved
   * ✔ MLM can be activated
* LOCKED
   * After cooling-off or manual lock
   * Funds committed
   * Refund restricted
________________


🔹 Activation Rule
IF payment_status == CONFIRMED
   THEN activate_seat = TRUE
        activate_MLM = TRUE
ELSE
   activate_seat = FALSE
   activate_MLM = FALSE
________________


🔹 Chargeback Detection
Trigger:
* Payment provider sends reversal/chargeback
________________


🔹 Chargeback Handling Workflow
ON chargeback_event:


SET payment_status = CHARGEBACK
SET account_freeze_status = TRUE
SET freeze_reason = "CHARGEBACK"


BLOCK:
   - MEC generation
   - MLM payouts
   - academic access (optional based on policy)


TRIGGER:
   recovery_workflow()
________________


🔹 Recovery Workflow
Steps:
1. Notify student
2. Calculate:
   * Outstanding amount
   * Admin/recovery fee
3. Await repayment
________________


🔹 Reactivation Rule
IF outstanding_amount == 0
   AND admin_fees_paid == TRUE
THEN
   account_freeze_status = FALSE
   payment_status = LOCKED
________________

⚙️ 4. System Behavior
________________


🔹 4.1 Pool Creation (Per Period)
period = YYYY-MM


opening_balance = last_period.closing_balance
inflow_amount = SUM(confirmed_payments × commission_rate)


available_pool = opening_balance + inflow_amount + adjustments


CREATE CommissionReserveAccount(period)
Note: confirmed_payments = only payments with payment_status = CONFIRMED (and not under chargeback/freeze).
________________


🔹 4.2 Eligibility Filter
Before calculating payouts:
FOR each participant:


IF payment_status != CONFIRMED OR account_freeze_status == TRUE
  OR academic_status != ACTIVE
  OR referral_status != ACTIVE
THEN
  eligible_amount = 0
ELSE
  eligible_amount = gross_amount
________________


🔹 4.3 Total Demand vs Pool
total_demand = SUM(eligible_amount for period)


IF total_demand <= available_pool
   THEN prorated_amount = eligible_amount
ELSE
   proration_factor = available_pool / total_demand
   prorated_amount = eligible_amount × proration_factor
________________


🔹 4.4 Hard Cap Enforcement
total_payout = SUM(prorated_amount)


ASSERT total_payout <= available_pool
System must block any attempt that violates this.
________________


🔹 4.5 Commit & Close
committed_payout = total_payout
closing_balance = available_pool - committed_payout


SET CRA.status = LOCKED → after approvals
SET CRA.status = CLOSED → after payouts executed
________________


🔹 4.6 Carry Forward (Optional)
IF carry_forward_enabled == TRUE
   next_period.opening_balance = current_period.closing_balance
ELSE
   next_period.opening_balance = 0
________________


🔹 4.7 Payout Gating
IF earning.prorated_amount >= min_payout_threshold
   THEN payout_status = APPROVED
ELSE
   payout_status = BLOCKED (carry forward optional)
________________


🔹 4.8 Chargeback / Freeze Impact
IF account_freeze_status == TRUE OR payment_status == CHARGEBACK
THEN
   exclude from pool inflow (if not yet counted)
   set earning.status = REJECTED or HOLD
   reverse/uncommit if not PAID
